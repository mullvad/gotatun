# 2026-02-17 Assured's audit of GotaTun

Between 2026-01-19 and 2026-02-15 [Assured Security Consultants](<https://www.assured.se/>)
performed a code audit of GotaTun.

The entire GotaTun implementation was in scope, with the exception of the command line interface
(`gotatun-cli`), the Mullvad specific DAITA code and the external dependencies.

The key conclusions from the audit report state that no major vulnerabilities were found.
There are some minor concerns regarding deviations from the WireGuard specification as well as
usage of dependencies maintained by a single developer. These are discussed in more detail below.

The full audit report can be found [next to this file](2026-02-17-Assured.pdf),
or on [Assured's website].

[Assured's website]: https://www.assured.se/publications/Assured_Mullvad_GotaTun_Code_Review_2026.pdf

## Overview of findings

This chapter will list the twelve findings from the report along with Mullvad's response to them.
The finding count per risk severity is:
* **Two** issues rated as *low*.
* **Nine** findings rated as *notes*.
* **One** *good* finding.

### 3.1 Nifty `size_must_be` sanity check (**GOOD**)

The audit report points out that the usage of compile time checks to ensure types have the expected
size is a good practice. This is something we are going to continue doing. One of the reasons we
use Rust is the fact that the language and ecosystem really encourage and allows catching so
many issues at compile time.

### 3.2 LFSR used to generate peer identifiers (**LOW**)

GotaTun did not generate random 32 bit peer identifiers in the way that the WireGuard specification
calls for. The audit report points out that this does not weaken the protection of the traffic in
the tunnel, and says that this could reveal information about the number of connected peers and
how many handshakes have been exchanged with each peer.

For an observer watching all traffic from the beginning, the LFSR weakness provides almost no
additional information. The observer already has:
- Total peer count from tracking unique IP:port combinations.
- Session duration from counting handshake initiations from given IP:port

The 8 bit counter in the LFSR index increments on every handshake. Under normal conditions they
happen every 120 seconds. But under connectivity issues or retransmissions, they can happen
more often. This makes the counter an unreliable source for tunnel uptime. Add to that that
the counter wraps at 256. This means it will reset at least every ~8.5 hours. This makes
the information about tunnel uptime derived from the counter ambiguous.

The fix for this, to implement the peer identifiers according to the WireGuard specification was
done in [#83](https://github.com/mullvad/gotatun/pull/83) and was part of the 0.3.0 release.

### 3.3 Padding of payload not according to WireGuard specification (**LOW**)

GotaTun did not pad the payload to a multiple of 16 bytes before encryption. WireGuard works fine
without doing this, but the specification states that implementations should pad in order to
complicate traffic analysis. The audit report points out that this is not a major cryptographic
issue, but that it would be recommended to implement the padding according to the specification.

The padding was implemented to work as described in the specification in
[#79](https://github.com/mullvad/gotatun/pull/79) and part of the 0.3.0 release.

Mullvad wants to point out that the protection that this padding provides is somewhat similar in
nature, but much less powerful than our DAITA functionality. Mullvad recommends anyone who includes
sophisticated traffic analysis in their threat model to consider enabling DAITA.

### 3.4 Endpoint address only updated on handshake initiation (**NOTE**)

GotaTun did not update the remote peer's IP address according to the specification. This prevented
GotaTun from handling clients roaming between different IPs as fast and smooth as it should.

The excellent roaming support in WireGuard is one of its selling points, so we fixed this in
[#91](https://github.com/mullvad/gotatun/pull/91), and it was included in the 0.4.0 release.

### 3.5 Sensitive data not protected in memory (**NOTE**)

The audit report points out that GotaTun stores encryption keys and other sensitive data in memory
with no extra protection. While there is no perfect way to protect against memory readout,
there are ways to minimize the risk that sensitive data ends in core dumps, swap or similar.

The `wireguard-go` implementation has none of these protections. However, they have a note
saying they would like to, but it's not really feasible in Go. The kernel implementation
does zero out important memory after use, and the entire situation around swap and core dumps is
very different in kernel space.

Since GotaTun is currently not worse off than `wireguard-go`, which is what we aim to mainly
replace, we do not treat this as a regression in security. But GotaTun should of course strive
to be as safe as possible, so Mullvad will look into adding memory protection techniques in the
future.

### 3.6 Crypto dependencies can be updated (**NOTE**)

The *lower version bounds* on the `x25519-dalek` and `chacha20poly1305` dependencies were set to
versions that had known vulnerabilities in them in GotaTun.

The vulnerable versions were not being used, neither in GotaTun itself nor in the Mullvad VPN app,
due to newer versions of these dependencies being pinned in the lockfiles.

The lower bounds on these dependencies were bumped to safe versions in
[#78 for `x25519-dalek`](https://github.com/mullvad/gotatun/pull/78) and
[#76 for `chacha20poly1305`](https://github.com/mullvad/gotatun/pull/76).
These fixes are included in the 0.3.0 release in order to protect users of the GotaTun library
from accidentally pulling in a vulnerable version.

### 3.7 Extensive usage of unsafe in crate tun (**NOTE**)

GotaTun uses the [`tun`](https://crates.io/crates/tun) crate for creation and management of virtual
tunnel interfaces. This is such a critical part of GotaTun that we asked the auditors to look at it
during the audit, even though dependencies in general were not in scope.

The audit report points out that the `tun` crate has a lot of `unsafe` Rust code, and very little
documentation around what invariants must be upheld for the code to be used correctly. The auditors
recommend that we go through the `tun` crate and try to make improvements to it that we upstream.

Mullvad uses the `tun` crate quite extensively in many of our Rust crates. We are aware that it is
a crate with some risks attached to it (significant `unsafe` usage and not very actively maintained).
We have provided patches to it upstream in the past, and we will try to do even more so in the
future, maybe with a higher focus on reducing unsafe code, if the maintainers accept it.

### 3.8 Complex non-working pre-allocation of packet buffers (**NOTE**)

GotaTun implements a memory pool for reusing packet buffers. The audit identified a bug where the
contiguous buffer intended for this pool was discarded and packets ended up being heap allocated,
potentially hurting performance.

The auditors were not able to find any security issues with this implementation. But they point
out that an overly complicated buffer allocation and reuse scheme has a higher risk of having
bugs in general.

This was fixed in [#75](https://github.com/mullvad/gotatun/pull/75) and included in the
0.3.0 release. The implementation was changed to remove the unused contiguous buffer and instead
simply allocate each packet on the heap. We will investigate if we can and should
simplify the allocation more than we already have.

### 3.9 Usage of the crate ring (**NOTE**)

GotaTun uses the `ring` crate to provide the underlying cryptographic primitives for
ChaCha20-Poly1305. The audit report points out that this is a one person project and that the
author describes `ring` as "An experiment". The auditors recommend replacing the `ring` crate,
or perform an audit of it and carefully monitor BoringSSL (which is the code `ring` builds upon)
for discovered vulnerabilities.

`ring` has for a long time been the de facto standard cryptographic primitive implementation in
the Rust ecosystem. Rustls (the main Rust native TLS implementation) has for example used
`ring` as the default cryptographic primitive provider until 2024. Even today (Feb 2026),
`ring` remains extremely well used, with roughly double the amount of daily downloads
compared to the main alternative, `aws-lc-rs`.

For ChaCha20-Poly1305 there are more or less two alternatives to `ring` that we can consider:

1. `chacha20poly1305`, a native Rust implementation. We benchmarked GotaTun with `ring`
   replaced by `chacha20poly1305`. The result was roughly a 50% drop in tunnel throughput.
2. `aws-lc-rs`, an API compatible drop-in replacement to `ring`. Maintained by AWS. Just like
   `ring`, this builds on top of code from BoringSSL. We benchmarked GotaTun with `aws-lc-rs`
   and saw equivalent performance compared to `ring`.

Given that `ring` has no known security issues, and is so well used in the entire ecosystem,
we do not see using this as an immediate risk. But we will consider either migrating to
`aws-lc-rs`, or allow the library user to select backend via cargo feature flags,
similarly to Rustls.

### 3.10 Simultaneous decryption and authentication (**NOTE**)

The GotaTun library uses `ring`'s `open_in_place` method to authenticate and decrypt data.
This implementation first decrypts the data, then it authenticates it. If the authentication
fails, the decrypted data is zeroed out so it cannot be accidentally used. This breaks the best
practice in cryptography to authenticate data before operating on it. `ring` has a
[comment](https://github.com/briansmith/ring/blob/7c0c4156f48164c8beda234215de862277afbf41/src/aead/algorithm.rs#L102-L111)
about this order not being the optimal one, but it is kept this way because some of the underlying
implementations interleave the decryption and tag computation for performance.

It's not a security issue per se to decrypt before authenticating. The practice of authenticating
first exists to make it harder to accidentally use the potentially malicious decrypted data for
anything. But as the API of `ring` prevents us from accessing the data in case authentication
fails, this is not really a practical problem.

Both the `aws-lc-rs` library and the Linux Kernel WireGuard implementation also decrypt the data
before verifying the tag. Given how common it is for respected and well used implementations to
ignore this best practice, we do not think this is a major concern.

The `chacha20poly1305` crate seems to implement this in the recommended way (authentication
first, then decryption). But since this crate lowers tunnel throughput by roughly 50%
and the security concern is only theoretical, we do not think it is a good idea to change
to this implementation.

### 3.11 About the validation of checksums (**NOTE**)

The GotaTun code has a number of `TODO` comments about implementing IP packet checksum verification
after decrypting incoming packets. Since the packet at this point has successfully passed the
authentication step of the decryption, we know it is intact. As the audit report states,
the WireGuard implementation in the Linux kernel also skips checksum verification and
explains why in a comment.

Mullvad will replace the `TODO` comments with an explanation on why
checksum validation is unnecessary.

### 3.12 Vulnerability in bytes (CVE-2026-25541) (**NOTE**)

GotaTun depended on a version of the `bytes` crate that had
[a vulnerability](https://rustsec.org/advisories/RUSTSEC-2026-0007.html) disclosed towards the end
of the audit period. GotaTun did not use `bytes` in any way that could trigger the vulnerability,
so this was not a problem in practice. Best practice is still of course to bump the dependency
to a version without the vulnerability in it.

This was fixed in [#82](https://github.com/mullvad/gotatun/pull/82) and part of the 0.3.0 release.
